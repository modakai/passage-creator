## Context

现有系统已经有文章创作流程、Human-in-the-Loop 阶段、文章 SSE 进度、Prompt 模板管理和 Prompt 使用日志。用户希望在标题生成完毕待选择、大纲生成完毕待编辑、正文融合完成三个环节收集提示词效果反馈，并在后台查看记录和四档满意度占比。

这个需求横跨用户创作页、Prompt 运行链路、后端持久化和管理后台。设计重点不是“加一个点赞按钮”，而是让反馈能稳定关联到任务、环节和提示词版本，否则统计结果无法指导后续 Prompt 迭代。

## Goals / Non-Goals

**Goals:**

- 在三个指定创作环节提供非强制、低打扰的反馈入口。
- 记录四档满意度评价、创作任务、环节、用户和提示词关联信息，并保留可选备注字段兼容后续扩展。
- 管理端支持反馈记录查询，并按环节、提示词模板和时间范围统计四档满意度数量与占比。
- 保持创作流程不因反馈缺失、提交失败或用户关闭反馈提示而阻塞。

**Non-Goals:**

- 不做强制满意度问卷，也不把反馈作为继续创作的前置条件。
- 不在本次变更中自动调整 Prompt 或实现 A/B 实验。
- 不要求用户对每一次创作都完整填写三次反馈。
- 不把跳过反馈的次数计入满意度占比分母；需要单独统计曝光/跳过时再设计埋点。

## Decisions

### Decision: 反馈环节使用稳定枚举

采用 `TITLE_SELECTION`、`OUTLINE_EDITING`、`CONTENT_MERGED` 三个反馈环节枚举，分别对应标题生成完毕待选择、大纲生成完毕待编辑、正文融合完成。

原因：前端展示文案和后端工作流阶段名称可能会变，统计维度不能依赖 UI 文案。枚举还能让接口校验、数据库索引和后台筛选保持一致。

备选方案是直接复用 `ArticlePhaseEnum`。该方案简单，但 `ArticlePhaseEnum` 描述的是创作流程状态，不完全等价于反馈采集点；未来流程拆分时容易把反馈统计维度带偏。

### Decision: 每个用户、任务、环节只保留一条有效反馈

反馈表对 `user_id + task_id + feedback_stage` 建唯一约束。用户重复点击时更新评价和备注，而不是插入多条同阶段反馈。

原因：用户误触或反复切换评价会严重污染满意度占比。统计要回答的是“这一环节这次产出的最终主观结果”，不是点击次数。

备选方案是完整记录每次点击。该方案适合行为分析，但当前目标是 Prompt 效果统计，重复记录会让管理端占比失真。

### Decision: 反馈记录关联 Prompt 使用结果快照

反馈记录存储 `prompt_usage_log_id`（可为空）、`prompt_template_id`、`template_key`、`version`、`environment` 等快照字段。创建反馈时优先按 `task_id + feedback_stage` 解析最近的相关 Prompt 使用日志，解析不到时仍允许提交，并记录可用于排查的任务和环节。

原因：Prompt 模板会发布、归档和更新，仅存模板 ID 不足以回溯当时用户评价的是哪一版提示词。保留快照能让统计在模板变更后仍可解释。

备选方案是只关联文章任务。该方案实现快，但后台看到“标题不好”时无法定位到具体 Prompt 版本，反馈价值大幅下降。

### Decision: 用户端反馈提交与创作流程解耦

前端在对应阶段渲染轻量反馈控件，反馈接口失败只提示用户，不改变标题确认、大纲编辑或正文完成状态。后端反馈服务不触发工作流 resume，也不更新文章业务状态。

原因：用户明确要求弱提示。反馈是观测能力，不是创作流程控制能力。

备选方案是把反馈字段塞进标题确认/大纲确认请求。该方案看似少一个接口，但会把“确认业务输入”和“评价 Prompt 效果”绑死，既增加用户负担，也无法覆盖正文融合完成后的独立反馈。

### Decision: 后台统计按已提交反馈计算四档满意度占比

管理端统计接口返回 `verySatisfiedCount`、`satisfiedCount`、`neutralCount`、`unsatisfiedCount`、`totalCount` 以及对应的四个占比字段，其中 `totalCount` 等于四档数量之和。跳过、关闭和未曝光不进入分母。

原因：当前需求没有要求曝光量埋点。把没有反馈的创作任务算作中性或负面都会制造错误信号。

备选方案是将未反馈计入总创作次数。该方案能体现反馈率，但不能代表 Prompt 满意度，需要另一个“反馈覆盖率”指标支撑，暂不纳入本次范围。

## Risks / Trade-offs

- [Risk] Prompt 使用日志与反馈环节无法可靠匹配 -> 先建立明确的 stage 到 templateKey/agentName 映射，匹配失败时允许反馈落库但在管理端标记为未关联。
- [Risk] 弱提示导致反馈量偏低 -> 管理端统计仅表达已提交反馈的四档满意度占比，不把它包装成全量用户满意度。
- [Risk] 用户重复提交改变历史统计 -> 采用同阶段 upsert，保留 `created_at` 和 `updated_at`，后台展示最后有效评价。
- [Risk] 反馈备注可能包含敏感内容 -> 管理端仅管理员可见，后端限制备注长度并沿用现有审计/脱敏策略。

## Migration Plan

1. 新增 Prompt 反馈数据表、枚举和必要索引。
2. 新增用户端反馈提交接口，并接入 Prompt 使用日志关联逻辑。
3. 新增管理端反馈记录分页和统计接口。
4. 前端创作页在三个指定阶段接入弱反馈控件。
5. 管理后台新增反馈记录/统计页面，并接入侧边栏入口。
6. 回滚时可隐藏前端入口和后台菜单；已落库反馈记录保留，不影响创作主流程。

## Open Questions

- 标题、大纲、正文融合当前是否都已经稳定记录 Prompt 使用日志；若正文融合没有对应模板日志，需要先补齐记录点，否则反馈只能关联到任务和环节。
- 管理端是否需要导出反馈记录；本次提案不包含导出，后续可单独扩展。
